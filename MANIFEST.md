# Manifest

## Зачем

Основная задача — **не переписать весь легаси**, а **зафиксировать уже готовые файлы** и **не дать им стать легаси при правках**.

Разработчик помечает файл `@laravel-ready`. При изменении — один прогон CLI на этот файл. **Git hook** отменяет коммит, если guarded-файл перестал соответствовать правилам.

Остальной код проекта можно не трогать: он вне зоны охраны, пока не получит метку.

## Что делаем

- Определяем **уровень готовности** файла к Laravel (шкала + причины, не просто `да/нет`).
- Детектим **блокеры легаси** в коде (суперглобали, `global`, `define()`, …) — см. `READINESS_MODEL.md`.
- **Охраняем** файлы с `@laravel-ready`: блокеры, метка `@legacy-code`, недопустимые зависимости → провал.
- Проверяем **прямые зависимости** (`use`) у guarded-файлов — зависимый класс должен быть `@laravel-ready` или `@adapter`.
- Отдаём **exit code** для git hooks и CI.

## Как используют

```
правка файла → laravel-ready path/to/File.php → exit 0 / 1
коммит       → pre-commit на staged .php      → тот же CLI
```

Полный прогон репозитория не обязателен. Типичный режим — **один файл** или **только staged** в хуке.

Git-хуки разработки пакета и продуктовый сценарий для легаси — `GIT_HOOKS.md`.

## Уровни и метки

**Уровни:** `Legacy` → `LaravelReady` или `LegacyPerfect`; дальше `LaravelPerfect`. Подробно — `READINESS_MODEL.md`.

**Метки** (ставит разработчик): `laravel-ready`, `legacy-code`, `legacy-perfect`, `adapter`, `for-legacy`.

`@laravel-ready` — opt-in: «этот файл под охраной». Без метки CLI **не блокирует** коммит (информирует или молчит).

## Простота

Одна команда, понятный вывод. Новые проверки — через правила, без правки ядра. Технические решения — `ARCHITECTURE.md`.

## Не делаем

- Массовую миграцию легаси в Laravel
- Генерацию кода, авто-рефакторинг
- Замену PHPStan / Psalm
- Автоматическое назначение меток
- Runtime-анализ
- Полный транзитивный граф зависимостей (на старте — только прямые `use`)
